# 项目级规范（Project Standards）

> **这是什么**：一份**持久的、跨 change 存活**的项目级约束文档。OpenSpec 的 change 用完即归档，
> 而本文件不归档——它是 AI 每次写代码（`apply`）和验收（`verify`）时必须遵守的边界与规范基线。
>
> **怎么长出来**：不要求一次写满。初次可只填已知项，其余留空；`design` 阶段每识别到一条
> **项目级**（而非 change 特定）的架构规则，就增量回填到这里。空小节保留标题即可。
>
> **填写原则**：只写**能约束 AI 行为的硬规则**（可据此判定"违规/合规"）。模糊的价值观
> （"代码要优雅"）不写——写不可判定的规则等于没写。

---

## 1. 分层与依赖方向

> 谁能依赖谁、禁止哪些方向的依赖。这是防止"模块互相渗透"的核心。

- 层次（由内到外 / 由稳定到易变）：<例：domain → application → infrastructure → interface>
- 允许的依赖方向：<例：外层可依赖内层；内层禁止依赖外层>
- 禁止的依赖：<例：domain 禁止 import 任何框架 / IO / 第三方 SDK>
- 跨模块调用方式：<例：只能通过显式接口 / 事件，禁止直接 import 对方内部实现>

## 2. 命名词汇表

> 同一概念在全项目用同一个名字。这是防止"AI 这次叫 UserService 下次叫 AccountManager"的核心。

| 概念 | 统一命名 | 禁止的别名 | 备注 |
|---|---|---|---|
| <例：用户账户> | <例：Account> | <例：User / UserInfo> | <例：领域内统一用 Account> |
|  |  |  |  |

- 命名约定：<例：接口用名词、用例用动词短语；文件名 kebab-case；类名 PascalCase>

- 目录结构约定：<例：测试镜像源结构 `src/x/y.ts` ↔ `test/x/y.test.ts`>

## 4. 禁止模式

> 明确不允许出现的写法。每条都应能在代码评审中被客观指认。

- <例：领域逻辑禁止写进 controller / 路由处理函数>
